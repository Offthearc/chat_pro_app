## Decisions

- The existing `DMThread`/`DMMessage` types (from the architect) were preserved in `src/types/index.ts` for backward compatibility. New `DMConversation` and `DMMessageNew` interfaces were added alongside them, matching the task specification exactly.
- `DMMessageNew` was named to avoid collision with the existing `DMMessage` type. In `src/api/dm.ts` and all new hooks/pages only `DMConversation` and `DMMessageNew` are used.
- The conversation id is `[user1Id, user2Id].sort().join(':')` so the id is always the same regardless of argument order.
- `sendDMMessage` increments `unreadCount` on every send (from either participant). `markRead` resets it to 0. This is a simple approximation — in a multi-user scenario unread tracking would be per-participant; for a two-person DM it is acceptable for MVP.
- The DMThreadPage navigates to `/dm` if the current user is not a participant in the conversation (guard against URL manipulation). In tests this required that conversations be created with the actual registered user objects (not hardcoded ones with different UUIDs).
- The `getConversations` call inside `DMThreadPage` is made synchronously during render (not in a hook) solely to look up the other participant's username for the header. This avoids an additional hook while keeping the component simple.
